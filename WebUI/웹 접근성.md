# 웹 접근성

모든 사용자가 모든 기기에서 웹에 접근할 수 있도록 하는 것
<br/><br/>

## 웹 접근성을 지켜야 하는 이유

1. 장애인 차별 금지법

    장애인도 비장애인과 동등하게 전자정보 및 비전자정보를 사용할 수 있도록 해야 한다는 내용. (2013년 4월 11일부터)

    <details>
	<summary>지켜지지 않았을 때</summary>
	
	1. 인권위의 민/형사 고발 → 형사: 3개월 이내 수사, 민사: 손해배상제도 절차, 악의로 인정될 경우 3년 이하 징역 또는 3천 만원 이하 벌금

	2. 인권위에서 진정 접수, 협의, 조정, 구제 조치 등 시정 권고, 법무부가 불이행 확인일로부터 3개월 내 시정할 것을 명령, 정당한 사유없이 명령 불이행 시 3천만원 이하 과태료
    </details>
2. 웹 창시자 팀 버너스 리는 "웹의 힘은 보편성에 있다. 장애에 구애 없이 모든 사람이 접근할 수 있도록 하는 것이 웹의 필수 요소다."라고 했다.

    즉, 웹 접근성이 바로 웹의 필수 요소인 것이다.
<br/><br/>

## 장애 환경

<details>
	<summary>웹 접근에 장애가 될 수 있는 환경</summary>
	
	1. 전맹 시각 장애

        시력이 전혀 없거나 거의 없다. 청각/촉각 같은 다른 감각으로 웹을 이용한다.

        청각적으로는 스크린리더(스크린을 읽어주는 프로그램)가 웹에 있는 이미지, 글씨 등을 음성으로 읽어준다.

        촉각적으로는 점자 정보 단말기로 점자를 손으로 읽으며 웹 페이지의 내용을 인식한다.

    2. 저시력 시각 장애

        안경, 렌즈, 치료, 수술로 해결할 수 없는 시력으로, 일상적인 생활이 어렵다.

        좀 더 크고 선명하게 볼 수 있는 기능을 이용한다.

        시야 장애의 경우, 페이지의 일부만 볼 수 있다. 이럴 땐 화면 확대 기능을 이용한다.

        색약, 색맹은 색상을 분별하는데 어려움이 있다. 더 선명하게 웹 페이지를 볼 수 있도록 고대비를 사용한다.

    3. 중증 운동 장애

        손 또는 팔을 사용하지 못하고, 목만 움직일 수 있는 장애 환경이다.

        보조기기(헤드 포인터, 빅키 키보드, 키가드)로 키보드를 조작한다.

        자판이 큼직한 빅키 키보드, 헤드 포인터를 머리에 고정하여 키보드를 조작할 수 있다. 키가드로 타이핑 스틱이 엇나가지 않고 보다 정확하게 타이핑한다.

    4. 손 운동 장애

        한 손만 사용 가능.

        한 손 사용자용 키보드와 트랙볼 마우스로 한 손으로도 마우스와 키보드를 쉽게 조작할 수 있다.

    5. 청각 장애

        들을 수 없기 때문에, 알림음이나 영상 같은 정보는 안내 문구나 자막 같은 화면의 글씨로 제공 받는다.

    6. 맥 사용자

        운영체제나 브라우저 같은 소프트웨어의 환경에 따른 문제도 있다.

        윈도우에서만 사용 가능한 웹은 맥에서 접근이 불가능하기 때문에, 맥이 장애 환경이 될 수 있다.

    7. 느린 인터넷

        인터넷이 느려 로딩이 잘 안 되어 UI가 깨지는 등의 경우는, 비장애인이 웹 페이지를 인식하는데 불편함을 줄 수 있다.
</details>

## 웹 접근성 지침

1. WCAG

    W3C에서 만든 웹 콘텐츠 접근성 지침

2. KWCAG

    WCAG를 토대로 국내에 맞게 반영된 한국형 웹 콘텐츠 접근성 지침

    원칙 4개, 지침 13개, 항목 24개로 구성. (항목은 아래 별도로 정리)
    
    | 원칙 | 지침 |
    | ---- | ---- |
    | 인식의 용이성 | 대체텍스트<br/>멀티미디어 대체 수단<br/>명료성 |
    | 운용의 용이성 | 입력장치 접근성<br/>충분한 시간 제공<br/>광과민성 발작 예방<br/>쉬운 내비게이션 |
    | 이해의 용이성 | 가독성<br/>예측 가능성<br/>콘텐츠의 논리성<br/>입력 도움 |
    | 견고성 | 문법 준수<br/>웹 애플리케이션 접근성 |

    장애인, 고령자 등의 정보 접근 및 이용 보장은 법적으로도 명시되어 있고, 2015년까지 한국의 모든 인터넷 사이트가 웹 접근성을 보장하도록 규정하고 있다.

## 웹 접근성 지침의 항목

### 1) 적절한 대체 텍스트 제공

텍스트가 아닌 콘텐츠는 스크린 리더가 읽어줄 수 없다. 그 의미나 용도가 무엇인지 그가 인식할 수 있도록 대체 텍스트를 제공해야 한다.

ex) 이미지

```html
<img src="example.png" alt="대체 텍스트">
```

비장애인이 이미지로 어떤 정보를 얻는 것과 동등하도록 대체 텍스트를 제공해야 한다. (스크린 리더가 alt값을 읽어준다.)

img 태그 외에도 input image나 button , map, area 태그 등에도 alt로 대체 텍스트를 제공할 수 있다.

대체 텍스트가 너무 긴 경우, alt를 빈 값으로 제공하고, 대체 텍스트는 따로 제공할 수도 있다.

(**alt 속성은 무조건 있어야 함**. alt 속성이 없거나 alt 속성 값이 공백(alt=" ")이면 src의 파일명을 읽기 때문. 대체 텍스트가 필요없거나 다른 것으로 대체되어도 alt 속성 값을 빈 값(alt="")으로 주어야 한다.)

```html
<img src="example.png" alt="">
<p class="hidden">
	대체 텍스트
</p>
```

스크린 리더는 요소에 따라 내용을 읽어주므로(예로, ul·li는 목록으로 읽음), 마크업을 구조적으로 하는 것이 중요하다. 예시에는 p 태그를 사용했지만, 대체 텍스트의 내용에 따라 다른 요소를 사용한다.

대체 텍스트 용으로 만든 마크업은 보이지 않도록 처리해야 한다.

사용자가 업로드하는 경우, 대체 텍스트를 입력하는 툴을 함께 제공해야 한다.

### 2) 자막 제공

동영상 같은 멀티미디어 콘텐츠에는 음성을 들을 수 없는 사용자들을 위해 대체 수단으로 자막, 대본, 수화 중 한가지 이상을 제공해야 한다.

소리 없이 텍스트만 나오는 영상의 경우에는 시각적으로만 인식이 가능하므로, 원고 등의 대체 수단이 있어야 한다.

사용자가 업로드할 땐 사용자가 대체 콘텐츠를 제공할 수 있는 툴을 제공해야 한다.

### 3) 색에 무관한 콘텐츠 인식

색을 구분하지 못하더라도 콘텐츠를 인식하는데 문제가 없어야 한다.

예시)

차트 → 패턴, 모양, 테두리 등으로도 구분

현재 보이는 슬라이드의 버튼 → 모양으로도 구분

현재 선택된 페이지내이션 버튼 → 테두리나 굵기 등으로도 구분

현재 선택된 탭 메뉴 → 배경 등으로도 구분

### 4) 명확한 지시 사항 제공

지시 사항은 모양, 크기, 위치, 방향, 색, 소리 등에 관계없이 인식될 수 있어야 한다.

예시)

회원 가입 폼에서 필수 입력 사항을 표시할 때 * 표시만 있으면 안된다. '* 표시는 필수 입력 사항입니다.'라는 설명이 제공되어야 한다.

왼쪽의 숫자를 입력하세요와 같은 위치나 방향, (대체 텍스트로 무엇이 작은 버튼인지 명시하고 있지 않았으면서) 작은 버튼을 클릭하세요와 같은 크기로만 가지고 지시하면 안된다.

### 5) 텍스트 콘텐츠의 명도 대비

텍스트 콘텐츠와 배경 간의 명도 대비는 4.5:1 이상이어야 한다.

명도 대비가 낮으면 가독성이 떨어진다.

이미지로 된 텍스트도 명도 대비를 준수해야 한다.

확대가 가능한 브라우저에서는 3:1 이상이어도 괜찮다. (그래도 되도록이면 4.5:1 이상이 되도록 구현하는 것이 가독성에 좋다)

로고, 장식 목적의 콘텐츠, 마우스나 키보드로 포커스 됐을 때 명도 대비가 커지는 콘텐츠는 예외로 한다.

### 6) 자동 재생 금지

자동으로 소리가 재생되지 않아야 한다.

콘텐츠의 소리가 자동으로 나오면, 스크린 리더의 음성과 겹쳐 내용 인식이 어렵다.

예전에 카페나 블로그에서 배경 음악이 자동으로 재생되게끔 했던 게 생각난다. 이러면 안된다고 한다.

음성이 나오지 않는 동영상은 자동 재생 금지 항목에 포함되지 않는다.

불가피하게 제공해야할 땐,

- 3초 내에 정지
- esc 키 선택 시 정지
- 소스 상 가장 먼저 제공하여 정지 기능 실행 가능하도록 구현

위 3가지의 해결 방안이 있다.

마우스 오버나 키보드 초점만으로 3초 이상의 배경음 콘텐츠가 재생되는 경우도 금지다. 클릭이나 선택 후에 재생이 되도록 구현해야 한다.

### 7) 콘텐츠 간의 구분

웹 페이지의 콘텐츠들은 각각 구분이 되어야 한다.

구분 방법

1. 테두리 이용
2. 구분선 삽입
3. 서로 다른 무늬 이용
4. 배경색 간의 명도대비(채도)로 구분
5. 줄 간격 및 글자 간격 조절
6. 기타 콘텐츠 이용

### 8) 키보드 사용 보장

모든 기능을 키보드만으로도 사용할 수 있어야 한다.

사용자의 반응 속도나 지속성이 중요한 붓질과 같이, 키보드 접근이 불가능한 기능은 예외다.

예시)

내비게이션 메뉴에 마우스 오버일 때만 서브 메뉴가 드롭 다운되는 경우, 마우스 사용이 불가한 키보드 이용자는 서브 메뉴에 접근할 수 없다. 키보드로 접근했을 때도 서브 메뉴가 뜨도록 해야 한다.

아래 코드처럼 키보드로 초점을 받을 수 없는 요소에 마우스 관련 이벤트가 적용됐을 때도 문제가 된다. 키보드만으로는 해당 이벤트를 실행할 수 없기 때문이다.

```html
<img onclick="click_event();">
```

→ a 태그나 버튼 태그로 키보드의 초점을 받을 수 있는 요소를 사용해야 한다.

(a태그에 href 속성이 없으면 키보드 접근이 불가능하다. a태그에는 href 속성을 꼭 넣어줘야 한다.)

아래처럼 a태그에 onfocus="this.blur();"를 사용하면 초점이 초기화되기 때문에, 키보드로 접근할 수 없게되므로, 사용하지 않아야 한다.

```html
<a href="#" onfocus="this.blur();"></a>
```

### 9) 초점 이동

키보드에 의한 초점은 일반적으로 좌→우, 상→하로 이동할 것이라고 예측한다.

스크린 리더 사용자는 화면은 보이지 않고, 소리로만 내용을 인식한다. 이때 읽는 순서를 중구난방으로 하면 내용 인식이 힘들 것이다.

마찬가지로 키보드 사용자도 화면의 순서와 다르게 중구난방으로 초점이 이동한다면 사용하기 어려울 것이다.

**초점 이동의 순서가 사용자가 일반적으로 예측할 수 있는 순서와 같아야 한다**.

초점의 순서를 tabindex로 조절이 가능하나, **기본적으로 마크업이 논리적으로** 이루어져 있으면 tabindex를 사용하지 않아도 된다.

레이어 팝업이 열리면, 초점은 팝업 내부로 이동하고, 팝업이 닫히면 원래 초점으로 돌아간다.

초점의 위치가 시각적으로 보여야 한다. hidefocus나 css의 outline:none이나 onfocus="this.blur();"를 사용하면 초점이 보이지 않으니 주의해야 한다.

### 10) 조작 가능

사용자 입력 및 컨트롤은 조작 가능하도록 제공되어야 한다.

컨트롤 하는 버튼이 너무 작으면, 마우스를 정교하게 사용할 수 없는 사용자에게 불편함을 줄 수 있다.

1. 컨트롤 대각선 길이 6mm 이상이 되어야 한다.
2. 테두리 안쪽으로 1픽셀 이상의 여백이 있어야 한다.

### 11) 응답 시간 조절

지적장애인이나 학습장애가 있는 경우, 문서를 읽고 이해하는데 더 많은 시간이 필요할 수 있다. 그러므로 **사용자가 시간 제한이 있는 콘텐츠의 응답 시간(최소 20초)을 조절할 수 있도록 해야 한다**. (ex. 시간 연장 기능)

(실시간 게임 같은 반응 시간 조절이 원칙적으로 허용되지 않는 경우는 제외)

사용자에게 시간 제한이 있음을 알리고, 종료되었을 때도 알려야 한다.

세션 시간이 20시간 이상인 콘텐츠는 예외.

시간 제한이 있는 콘텐츠는 가급적 포함하지 않는 것이 바람직하다.

### 12) 정지 기능 제공

자동으로 변경되는 콘텐츠의 움직임을 제어할 수 있어야 한다.

예를 들어, 자동 슬라이드 같은 경우에 정지, 이전, 다음 기능을 제공해야 한다.

마우스 오버나 키보드 접근 시 정지하는 것도 정지 기능을 제공한 것으로 인정된다.

실시간 검색어처럼 자동 변경되는 콘텐츠에, 마우스 오버나 키보드 접근 시 모든 콘텐츠가 노출되는 경우도 인정된다.

즉, **자동으로 변경되는 콘텐츠에는 정지, 이전, 다음 기능을 제공하거나 모든 콘텐츠를 보여주는 기능을 제공해야 한다.**

### 13) 깜빡임과 번쩍임 사용 제한

초당 3~50회 주기로 깜빡이거나 번쩍이는 콘텐츠를 제공하지 않아야 한다.

밝은 빛의 화면 점멸이 연속적으로 나오는 장면은 발작을 일으킬 수 있다. (실제 사례 - 포켓몬 쇼크)

해결 방안)

- 번쩍이는 콘텐츠가 차지하는 면적의 합이 화면 전체 면적의 10% 미만
- 시간을 3초 미만으로 제한
- 사전에 경고하고 중단 가능한 수단을 제공

비장애인에게도 빠르게 깜빡이는 화면은 눈에 피로를 줄 수 있기 때문에, 가능한 제공하지 않아야 한다.

[PEAT](https://trace.umd.edu/peat/) - 콘텐츠가 초당 3~50회 깜빡이는지 확인할 수 있는 프로그램

### 14) 반복 영역 건너뛰기

콘텐츠의 반복되는 영역은 건너뛸 수 있어야 한다.

예를 들어, 검색창이나 내비게이션 메뉴 같은 경우에는 여러 페이지에서 반복되어 나온다. (이런 영역을 반복 영역이라고 한다) 스크린 리더는 이런 영역을 매 페이지마다 읽어준다. 스크린 리더 사용자에겐 페이지를 이동할 때마다 같은 걸 반복해서 들어야 하는 불편함이 있다.

그래서 반복 영역이 있으면 이를 읽는 것을 건너뛸 수 있는 기능을 웹 페이지 가장 앞에 제공해야 한다.

건너뛰기 기능은

- 마크업상 최 상단에 위치해야 한다.
- 건너뛰기로 인해 도착하는 영역은 같은 페이지 내에 존재해야 한다.
- 키보드 접근이 가능해야 한다.
- 접근 시 화면에 노출되어야 한다.
- '하단 메뉴로 바로가기'와 같은 위치 정보 제공은 부적절하다.

반복 영역이 여러개일 경우, 메인인 본문으로 이동하는 건너뛰기 링크를 제일 먼저 나타내야 한다.

### 15) 제목 제공

페이지, 프레임, 콘텐츠 블록에는 적절한 제목을 제공해야 한다.

- 페이지 제목

```html
<html lang="ko">
	<head>
		<title>제목</title>
...
```

웹 페이지를 켰을 때 브라우저의 탭에 뜨는 제목인데, 이 제목으로 해당 페이지의 핵심 내용을 제공해주면 된다.

- 프레임 제목

프레임이나 아이프레임에도 각 프레임에 맞는 제목을 제공해야 한다.

```html
<iframe title="제목" ...></iframe>
```

내용이 없는 아이프레임도 제목으로 "빈 프레임"이나 "내용 없음"으로 제공해야 한다.

- 콘텐츠 블록 제목

콘텐츠 블록에도 제목을 주어야 하는데, 같은 depth의 제목은 같은 h 태그를 사용해야 한다. 예를 들어 한 콘텐츠 블록의 제목에 \<h3>태그를 사용했으면, 그와 같은 depth의 제목에도 \<h3>를 사용해야 한다. (만약 제목이 보일 필요가 없으면 숨김 처리를 한다. 단, 스크린 리더에는 읽혀야 한다.)

- 특수기호

제목에 특수기호는 1개까지만 사용해야 한다. 만약 제목에 ★★★와 같이 쓴다면, 스크린 리더는 이를 '검은 별표, 검은 별표, 검은 별표'라고 읽기 때문이다.

### 16) 적절한 링크 텍스트

링크 텍스트는 용도나 목적을 이해할 수 있도록 제공해야 한다.

```html
<a href="#" title=""></a>
```

이렇게 빈 링크를 스크린 리더 사용자는 무슨 링크인지 알 수 없고, 이 링크에 따로 크기를 주지 않았으면 키보드 사용자는 초점의 위치를 알 수 없다. → 이런 빈링크는 제거한다.

이미지 링크 같은 경우에는 다음과 같이 이미지에 alt 속성 값을 꼭 넣어주어야 한다. 만약 alt 속성이 비어있으면, 링크 텍스트가 없는 것이다.

```html
<a href="#">
	<img src="btn_next.png" alt="다음 콘텐츠 보기">
</a>
```

css background-image를 이용해서 배경 이미지로 링크의 이미지를 제공한 경우에도 대체 텍스트를 제공해야 한다.

```html
<a href="#" class="link_next">다음 콘텐츠 보기</a>
```

```css
.link_next {
	background-image: url("./btn_next.png");
}
```

대체 텍스트 항목에서처럼 썸네일 이미지의 설명과 페이지에 보이는 제목이 같을 때는 이미지의 alt 속성값은 비워주고, a 태그를 하나로 감싼다.

```html
<a href="#">
	<img src="thumbnail.png" alt="">
	<span>제목</span>
</a>
```

틀린 예시

```html
<a href="#">
	<img src="thumbnail.png" alt="제목">
</a>
<a href="#">
	<span>제목</span>
</a>
```

또한, 링크 텍스트를 제공할 땐, 해당 링크의 명확한 목적지를 명시하는 것이 좋다. 더보기나 자세히 보기 같은 경우에도 어떤 것의 더보기나 자세히 보기인지를 함께 명시해주는 것이 좋다. (ex. 더보기 → 공지사항 더보기)

### 17) 기본 언어 표시

주로 사용하는 언어를 명시해야 한다.

스크린 리더는 선언된 언어에 따라 발음을 하기 때문이다.

ISO639-1에서 지정한 두 글자로 된 언어 코드이다.

한국어는 ko, 영어는 en이다.

설정 방법은 문서 타입에 따라 달라진다.

- HTML 4.01 / HTML5 표준

```html
<html lang="ko">
```

- XHTML 1.0 표준

```html
<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="ko" lang="ko">
```

- XHTML 1.1 표준

```html
<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="ko">
```

### 18) 사용자 요구에 따른 실행

사용자가 의도하지 않은 기능(새 창, 초점에 의한 맥락 변화 등)은 실행되지 않아야 한다.

그래서 요즘은 새 창(팝업) 대신 레이어 팝업으로 제공하고 있다. 단, 화면을 가리는 레이어 팝업은 여전히 오류이다.

- 반복 영역 건너뛰기보다 먼저 팝업을 제어할 수 있도록 구현하거나 가장 상단에 화면을 가리지 않도록 레이어 팝업을 제공해야 한다.
- 스크린 리더 사용자나 키보드 사용자를 위해 title이나 보이지 않는 텍스트로나 target="_blank"로 새 창이 뜰 것을 인지할 수 있도록 해야 한다.

```html
<a href="…">이용약관<span class="hidden">새 창</span></a>
<a href="…" title="새 창">이용약관</a>
<a href="…" target="_blank">이용약관</a>
```

window.open으로 새 창을 열면 인지할 수 없다는 점을 주의해야 한다.

(센스 리더 외의 다른 스크린 리더는 title이나 target="_blank"가 있어도 새 창이라고 읽어주지 않기 때문에, **텍스트를 따로 넣어주는 것이 확실한 해결 방안**이다.)

- 컨트롤 선택 시 바로 기능이 실행되지도 않아야 한다. 예를 들어, "오늘 하루 이 창을 열지 않음"이라는 체크박스를 선택하자마자 팝업이 닫히면 안 된다는 것이다. 체크박스 선택 후 닫기 버튼을 클릭해야 닫히도록 구현해야 한다.
- select의 옵션을 선택하자마자 페이지 이동하는 것도 문제가 된다. 아래 예시처럼 작성하면 onchange의 함수 때문에 옵션을 선택하면 바로 페이지 이동이 될 것이다. 또한, 키보드로도 접근하자마자 페이지 변경이 되어, 이후 콘텐츠에 키보드 접근이 불가하고, 8) 키보드 사용 보장에도 영향을 미친다.

```html
<select title="관련 사이트" onchange="myFunction(this.index)">
	<option value="www.xxx.xxx">...</option>
</select>
```

onfocus나 onkeypress 등의 이벤트로 포커스를 받는 것만으로도 새 창이나 레이어가 뜨거나 페이지가 바뀌는 모든 경우가 사용자가 원하지 않는 기능이 실행되는 것이다.

**사용자가 의도하지 않은 기능이 실행되지 않도록 구현해야 한다.**

### 19) 콘텐츠의 선형 구조

콘텐츠는 논리적인 순서로 제공해야 한다.

눈으로 보이는 순서가 아닌, 내용의 순서대로 마크업을 해야 한다.

예를 들어, 상단에 탭 메뉴가 나오고, 하단에 내용이 나오는 화면일 때, 탭 메뉴와 내용을 따로 작성해도 보기에는 이해하는데 문제가 없다.

하지만 스크린 리더로 읽을 땐 파악하기가 어렵다.

따라서 메뉴>내용>메뉴>내용 순으로 마크업하거나

메뉴끼리, 내용끼리 마크업을 하되, 각 내용에 메뉴 제목을 제공해주어야 한다.

### 20) 표의 구성

표는 이해하기 쉽게 구성해야 한다.

화면을 볼 수 있는 사용자는 이해하기 쉬운 표이지만, 스크린 리더를 사용하는 볼 수 없는 사용자들은 내용을 들으며 인식하기 때문에, 주의해야 한다.

제목 셀과 내용 셀을 각각 th, td로 마크업해서 구분해준다.

scope으로 행제목인지 열제목인지도 구분해주어야 한다.

제목과 요약 정보도 필요하다. 제목과 요약 정보는 달라야 한다.

XHTML, HTML4는 table의 summary 속성으로 요약 정보를 제공했지만, HTML5부터는 caption태그에 제목과 요소를 모두 제공한다.

복잡한 표의 경우엔 id와 headers로 나타내주면 좋다. 데이터를 읽어줄 때 해당하는 제목을 연결해서 읽어주기 때문이다.

데이터 테이블이 아닌, 레이아웃 테이블은 table이 아닌 다른 것으로 마크업하는 것이 좋다. 만약 table로 마크업한다고 해도 caption, summary, th는 제공하지 않아야 한다. 스크린 리더가 데이터 테이블로 착각할 수 있기 때문이다.

### 21) 레이블 제공

사용자 입력에는 대응하는 레이블을 제공해야 한다.

레이블이란, 사용자 입력의 용도나 역할에 대한 설명을 말한다.

입력 서식에 초점이 갔을 때, 스크린 리더는 편집창이라고만 읽어준다. 무엇을 입력해야 하는지 알 수 없기 때문에, label로 알려주어야 한다.

label의 for와 입력창의 id값을 동일하게 표시해주면 된다.

```html
<label for="user_id">아이디<label>
<input type="text" id="user_id">
```

또한, label을 선택해도 해당 입력창에 포커스가 가기 때문에, 편리하게 작성할 수 있다.

label을 눈에 보이게 하고 싶지 않으면 title 속성값을 주면 된다. (중복 사용은 x)

```html
<input type="text" title="아이디">
```

### 22) 오류 정정

입력 오류를 정정할 수 있는 방법을 제공해야 한다.

입력 서식을 잘못 입력한 체 제출했을 때, 모든 입력 내용이 사라지면 안된다.

입력 오류가 발생했을 때, 원인을 알려주고, 해당 서식으로 초점이 이동되어야 한다.

### 23) 마크업 오류 방지

마크업 언어의 요소는 열고 닫음, 중첩 관계 및 속성 선언에 오류가 없어야 한다.

- 열고 닫음 오류

```html
<ol> <!-- ol 태그가 닫히지 않음 -->
    <li>첫 번째 목록</li>
    <li>두 번째 목록</li>
    <li>세 번째 목록</li>

<ul>
    <li>첫 번째 목록</li>
    <li>두 번째 목록</li>
    <li>세 번째 목록</li>
</ul>
```

- 중첩 관계 오류

```html
<!-- a태그가 열리고 span 태그가 열렸으니,
span 태그를 먼저 닫고 a 태그를 닫아야 한다. -->
<a href="…"><span>바로가기</a></span>
```

- 중복된 속성 오류

```html
<!-- class 속성이 두번 적혀있다. -->
<img src="apple.jpg" alt="" class="img_box" class="img_01">
```

- id 속성 값 중복

```html
<!-- 하나의 마크업 문서에 같은 id 값이 있으면 안 된다.
레이블과 매칭된 서식의 id가 중복된 경우는 레이블 제공 항목에도 문제가 생긴다. -->
<div id="main_menu" class="lnb">
…
</div>

<ul id="main_menu" class="menu">
    <li>로그인</li>
    <li>회원가입</li>
</ul>
```

### 24) 웹 애플리케이션 접근성 준수

콘텐츠에 포함된 웹 애플리케이션은 접근성이 있어야 한다.

웹 애플리케이션(ex. 플래시)은 웹 콘텐츠에 포함되어 특정한 기능을 수행하도록 구성된 소프트웨어의 일종이다. 리치 인터넷 애플리케이션(RIA : Rich Internet Application)이라고도 한다.

이에 대한 접근성 프로그래밍 인터페이스 사용을 제공하거나 대체 수단 혹은 보조 기술을 지원해야 한다.

예를 들어, 접근성이 어려운 플래시의 경우엔 HTML 버전을 선택할 수 있는 대체 수단이나 대체 텍스트를 제공해야 한다.

## 웹 접근성 진단 도구

웹 접근성 지침들을 기준으로 접근성을 지켜쓴지 진단하는 도구들이 있다.

대부분 수동으로 진단을 하지만, 그 중 많이 사용하는 몇 가지 진단 도구들이다.

### Colour Contrast Analyser (CCA)

- 전경 색과 배경색의 명도 대비를 체크해주는 프로그램
- 측정하고 싶은 텍스트의 색과 배경 색을 선택하여 명도 대비를 확인할 수 있다.
- 명도 대비는 최소 3:1 이상이 되도록 구현해야 한다.
- [Color Contrast Analyser]<https://developer.paciellogroup.com/resources/contrastanalyser/>{:target="_blank"}

### KWCAG a11y inspector

- 컨트롤의 대각선 길이를 측정해주는 프로그램
- 크롬 웹 스토어에서 설치 가능
- 크롬에서 해당 프로그램의 아이콘을 클릭하여 실행한다.
- 컨트롤 크기는 대각선 6mm 이상으로 구현해야 한다.

### OpenWAX

- 접근성을 자동으로 진단해주는 프로그램
- 크롬 웹 스토어에서 설치 가능
- 자동 진단이 정확하지는 않아 수동 진단 병행 필요

### W3C Validation

- 웹 표준을 검사하는 사이트
- 접근성 오류, 특히 마크업 오류 확인 시 많이 사용

## 느낀점

이미 알고 있는 내용도 있지만, alt에 공백 값을 주면 src 값을 읽는다는 등의 디테일한 내용을 추가로 알게 되었다.

이미 찾아보거나 사용해본 내용이어도, 개발하면서 놓치게 되는 부분도 있고, 매번 할 때마다 적용 방법을 찾아보는 경우도 있다.

이번에 이렇게 강의를 보고 정리하면서 웹 접근성에 대한 디테일한 정보들도 공부해두고, 직접 써보기도 하면서 좀 더 자유자재로 웹 접근성에 맞는 마크업을 할 수 있게 되는 것이 목표다.
